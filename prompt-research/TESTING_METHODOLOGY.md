# Prompt Research Testing Methodology

## Date: 2026-03-13
## Author: Erik + Claude (Opus 4.6)
## Inspired by: [Karpathy's autoresearch](https://github.com/karpathy/autoresearch)

---

## 1. Goals

We have a simulated workplace sitcom with 5 characters who chat in a group chat about muffin pan recipes across a 7-day production cycle. The dialogue is generated by LLMs. Our goals:

1. **Optimize prompt quality** - Find the voice guides, rules, examples, and system prompt template that produce the most natural, distinct, conflict-rich dialogue
2. **Solve context degradation** - By Sunday (day 7), characters have no memory of Monday-Friday. Fix this through compression/summarization of prior days
3. **Do it autonomously** - Use an automated loop that modifies prompts, scores results, and keeps/discards changes without human intervention
4. **Keep costs low** - Use cheap models (Haiku) for generation, expensive models (Sonnet) for judgment, and auto-stop when improvements plateau

---

## 2. The Autoresearch Pattern

Based on Karpathy's approach: one mutable file, one locked evaluator, an autonomous loop.

### Architecture

```
[Mutable File] -- agent modifies --> [Generation Engine] -- produces --> [Evaluator] -- scores --> keep/discard
     ^                                                                                               |
     |                                                                                               |
     +-----------------------------------------------------------------------------------------------+
```

### Key Components

| Component | File | Role |
|-----------|------|------|
| Mutable prompts | `prompts.py` | Voice guides, shared rules, examples, system prompt template |
| Mutable compression | `compression_prompts.py` | Compression template, injection format, config |
| Generation engine | `scripts/simulate_dialogue_week.py` | Produces dialogue (LOCKED during experiments) |
| Evaluator | `evaluator.py` | QA scorer + LLM judge (LOCKED during experiments) |
| Prompt runner | `runner.py` | Loop for prompt optimization |
| Compression runner | `compression_runner.py` | Loop for compression optimization |
| Agent instructions | `program.md` / `compression_program.md` | Tells the agent what to modify and how |

### The Loop (per iteration)

1. Agent reads current best file + recent results
2. Agent suggests ONE modification (returns complete modified file)
3. Modified file is written and monkey-patched into the generation engine
4. Dialogue is generated for the target day
5. Dialogue is scored (QA + Judge)
6. If score > best: KEEP (save as new best). If not: DISCARD (restore previous best)
7. Results logged to TSV

## 3. Scoring System

### QA Score (0-100) - Structural Quality

Free, local Python evaluation. 16 categories including:
- **AI tells** - Em dashes, curly quotes, en dashes (HARD FAIL = 0)
- **Repetition** - Cross-character phrase repetition, self-repetition
- **Prompt echo** - Characters leaking system prompt language
- **Voice distinction** - Word count variance, vocabulary diversity across characters
- **Participation** - Balance of speaking turns
- **Conflict** - Presence of disagreement and tension
- **Verbosity** - Penalty for messages over 50 words
- **Name usage** - Penalty for formal addressing ("Thanks, Margaret")

### Judge Score (1-10) - Semantic Quality

LLM evaluation (Sonnet 4.6). Evaluates:
1. CHARACTER VOICE - Do characters sound distinct and consistent?
2. TENSION - Is there authentic disagreement or friction?
3. SPECIFICITY - Are details concrete and recipe-relevant?
4. NATURALNESS - Does it read like real coworkers, not AI?
5. CONTINUITY - Does it connect to prior days (if any)?
6. BOOKENDS - Does the day open and close naturally?
7. HALLUCINATIONS - Any fabricated ingredients, techniques, or references?

Score ranges: 1-2 Broken, 3-4 Weak, 5-6 Acceptable, 7-8 Good, 9-10 Excellent.

### Combined Score

```
combined = 0.6 * QA + 0.4 * (Judge * 10)
```

60% structural (cheap, consistent) + 40% semantic (expensive, nuanced). Judge is scaled from 1-10 to 0-100 before blending.

### Why Two Scores

QA and Judge measure different things. Experiment 33 on Monday scored QA=97 but Judge=3 (combined 70.2). High structural quality does NOT guarantee good dialogue. The reverse also happens - exp 24 on Monday got QA=94, Judge=7 (combined 84.4). Both signals are needed.

### Judge Frequency

Running the judge on every iteration is expensive. We use `judge_every=N` to only run it every Nth KEEP. Early iterations (first 3-4) always get judged to establish a baseline. The runner also runs judge-free (QA-only) mode for fast iteration.

## 4. Trend Detection and Auto-Stop

### Why We Need It

Without auto-stop, the loop burns through all iterations even when improvements have plateaued. Most runs find their best score in the first 5-15 iterations, then oscillate.

### How It Works

At checkpoint intervals (5, 10, 20, 50, 100 iterations), the runner analyzes the score history:

- **Slope** - Average score change per iteration over the last 5 scores
- **Sign changes** - How many times the score direction reversed in recent iterations
- **Since best** - How many iterations since the last improvement

### Trend Classification

| Trend | Condition | Action |
|-------|-----------|--------|
| **Climbing** | Positive slope, recent improvements | CONTINUE |
| **Plateau** | Near-zero slope, no recent improvement | CONTINUE (give it more room) |
| **Oscillating** | Frequent direction reversals | STOP if no improvement in 2x window |
| **Declining** | Negative slope, best was long ago | STOP |

### Auto-Stop Logic

Stop early when:
- `oscillating` AND no improvement in `window * 2` iterations
- `declining` (best was more than `window * 2` iterations ago)
- No improvement in `window * 3` iterations (regardless of trend)

### Results

| Run | Target | Max Iter | Stopped At | Best At | Best Score | Reason |
|-----|--------|----------|------------|---------|------------|--------|
| Monday (QA-only) | monday | 100 | Did not stop | exp 2 | QA=96 | Only 2 iterations before done |
| Monday (full) | monday | 100 | 50 | exp 11 | Combined=96.0 | Oscillating, 35 since best |
| Tuesday | tuesday | 100 | 20 | exp 4 | Combined=94.0 | Oscillating, 16 since best |
| Thursday | thursday | 100 | 50 | exp 34 | QA=97 | Declining, 11 since best. Hit API overload (~30 iterations lost) |
| Compression (QA-only) | sunday | 5 | n/a | exp 3 | QA=95 | Test run, only 5 iterations |
| Compression (full) | sunday | 100 | 20 | exp 5 | Combined=92.0 | Declining, 16 since best |

## 5. Frozen History and Caching

### The Problem

To test Tuesday's prompts, we need Monday's conversation as context. But if we regenerate Monday each iteration, we're measuring Monday variance, not Tuesday prompt quality.

### The Solution: Frozen History

1. Generate all days BEFORE the target day once, using original (unpatched) prompts
2. Cache to `baselines/{concept-slug}/frozen-before-{day}.json`
3. Every experiment iteration uses the SAME frozen history
4. Only the target day is regenerated with modified prompts

### For Compression Testing

Compression tests need a full Mon-Sat week frozen:
- Cached at `baselines/{concept-slug}/frozen-week-mon-sat.json`
- 36 messages across 6 days for "Jalapeño Corn Dog Bites"
- Generated once, never changes during experiments
- The ONLY variable is how these conversations get compressed into Sunday's context

### Why This Matters

Isolating the variable is everything. Without frozen history:
- Tuesday might score high because Monday happened to mention something useful
- Thursday might score low because Wednesday went in a weird direction
- You can't tell if the prompt change helped or if you just got lucky

With frozen history, the ONLY thing changing between iterations is the experimental prompt.

## 6. Test Types

### Type 1: Prompt Optimization (runner.py)

**Goal:** Find the best voice guides, rules, examples, and system prompt template for a specific day.

**What varies:** `prompts.py` (CHARACTER_VOICE_GUIDES, SHARED_CHARACTER_RULES, CHARACTER_EXAMPLE_MESSAGES, build_system_prompt())

**What's frozen:** All days before target, evaluator, generation engine

**Process:** Sequential day optimization. Optimize Monday, apply wins, then optimize Tuesday with frozen Monday, etc.

**Run command:**
```bash
PYTHONPATH=. doppler run --project muffinpanrecipes --config dev -- \
  uv run prompt-research/runner.py \
  --concept "Jalapeño Corn Dog Bites" \
  --target-day monday \
  --iterations 100 \
  --auto-stop
```

### Type 2: Compression Optimization (compression_runner.py)

**Goal:** Find the best way to summarize prior days' conversations so Sunday characters have useful context.

**What varies:** `compression_prompts.py` (COMPRESSION_TEMPLATE, COMPRESSION_SYSTEM_PROMPT, INJECTION_TEMPLATE, compress_day(), format_highlights())

**What's frozen:** Full Mon-Sat conversations, evaluator, generation engine, voice prompts

**Process:**
1. Freeze Mon-Sat once (36 messages)
2. Baseline: Sunday with NO compression (no context from prior days)
3. Initial: Sunday with default compression template
4. Loop: Agent modifies compression template -> compress all 6 frozen days -> generate Sunday with compressed context -> score -> keep/discard

**Run command:**
```bash
PYTHONPATH=. doppler run --project muffinpanrecipes --config dev -- \
  uv run prompt-research/compression_runner.py \
  --concept "Jalapeño Corn Dog Bites" \
  --iterations 100 \
  --auto-stop
```

### Future: Type 3 - Per-Day Adaptive Compression

**Not yet built.** Phase 2 of compression research.

**Goal:** Different compression strategies for different days based on content characteristics (contentious days get less compression, agreeable days get more).

**What varies:** Per-day compression rules within `compression_prompts.py`

**What's frozen:** Same Mon-Sat conversations as Type 2

## 7. The Context Window Problem (and Why It's Not What You Think)

### Symptom

Sunday dialogue is consistently the worst day. Characters hallucinate details (e.g., "brown butter" in a jalapeño corn dog recipe), lose continuity with earlier days, and repeat themes.

### Root Cause

It's NOT a model context window limit. Haiku has 200K tokens. The problem is in our code:

```python
# simulate_dialogue_week.py, line 656
history_depth = 8 if day_turn == 1 else 4
history = "\n".join(recent_lines[-history_depth:])
```

Each character only sees the last 4-8 messages in their prompt. By Sunday, only Saturday's messages are visible. Monday through Friday is simply not sent to the model.

### The Fix: Highlights Injection

Two implementations:

1. **Production (inline):** `_generate_day_highlights()` in simulate_dialogue_week.py generates a summary after each day, accumulates in `week_highlights`, and injects into every subsequent day's prompt
2. **Research (external):** `compression_runner.py` compresses frozen conversations and passes highlights via the `initial_highlights` parameter to `run_simulation()`

Both inject a "WEEK SO FAR" / "EARLIER THIS WEEK" block into the user prompt before the recent chat.

### Who Is In Each Meeting

Not all 5 characters participate every day:

| Day | Stage | Participants |
|-----|-------|-------------|
| Monday | Brainstorm | Margaret, Marcus, Steph |
| Tuesday | Recipe Dev | Margaret, Steph, Marcus |
| Wednesday | Photography | Julian, Steph, Margaret |
| Thursday | Copywriting | Marcus, Steph, Margaret |
| Friday | Final Review | ALL FIVE |
| Saturday | Deployment | Devon, Margaret |
| Sunday | Publish | Steph, Margaret, Marcus |

When Sunday characters reference "what Julian said Wednesday," it works because Steph and Margaret were there too - they carry context across days. The highlights injection enables this by including attribution ("Julian proposed dark backgrounds for the shots").

## 8. Key Findings

### Prompt Optimization Findings

1. **"Never address someone by name"** - Single biggest QA improvement. Characters were writing "Thanks, Margaret" constantly.
2. **"NEVER use em dashes or curly quotes"** - Hard QA fail prevention. Added to shared rules.
3. **System prompt section reordering** - Moving INTERNAL CONTRADICTIONS before HOW YOU SPEAK improved character distinctiveness.
4. **"Conflict is natural. Disagree when your character would disagree."** - Improved tension scores.
5. **"React to the specific food being discussed."** - Reduced generic dialogue.
6. **Wins generalize across days** - Tuesday optimization found no new improvements beyond Monday's wins.

### Compression Findings

1. **Compression helps** - QA-only: Sunday went from 90 (no context) to 95 (with compression). +5 points.
2. **Bad compression is worse than none** - With judge scoring, initial compression (67.4) scored LOWER than no compression (81.6). The judge caught awkward callbacks.
3. **"Still open" pattern works** - Flagging unresolved disagreements gives Sunday characters natural hooks to close threads.
4. **Hallucination guard is critical** - "Do NOT invent details that aren't in the summaries above" was a key winning change.
5. **Explicit callback requirements backfire** - "At least two characters must make explicit callbacks" sometimes produced forced-sounding references.
6. **QA and Judge diverge on compression** - QA loves compression (more specific content = higher scores). Judge is pickier about naturalness.

### Context Depth Findings (Run Comparison)

Three compression runs tested different context configurations for Sunday:

| Run | Context Config | Baseline | Best Score |
|-----|---------------|----------|------------|
| 1 | 8/4 depth, no Saturday seed | 81.6 | 92.0 |
| 2 | 12/8 depth, Saturday seeded | 83.8 | 96.0 |
| 3 | Full context (no truncation), Saturday seeded | 80.4 | 87.0 |

**Key insight:** Full raw context (all ~36 messages) performed WORST. Characters got distracted by volume rather than focusing on key moments. The 12/8 depth with Saturday seeding and compression highlights was the sweet spot — enough context for continuity but curated enough to stay focused. This validates the compression approach: curated highlights > raw message dump.

### API Reliability

The Thursday run hit Anthropic API overload (529 errors) for ~30 consecutive iterations. Without retry logic, all were logged as ERROR/skipped. After adding exponential backoff (5s, 10s, 20s), the compression test recovered from 3 errors seamlessly.

## 12. Retry Logic

Added after the Thursday API outage. Both runners use `retry_on_api_error()`:

- Catches 500 (Internal Server Error) and 529 (Overloaded) responses
- Up to 3 retries per call
- Exponential backoff: 5s, 10s, 20s
- If all retries fail, the iteration is logged as ERROR and skipped (same as before)
- Wraps: agent modification calls, compression calls, generation calls, evaluation calls

### Experiment: Progressive (Rolling) Compression — TESTED 2026-03-14, DID NOT WIN

**Hypothesis:** Progressive compression — where Monday's summary rolls into Tuesday's,
Mon+Tue rolls into Wednesday's, etc. — produces better Sunday dialogue than independent
per-day compression because it naturally preserves important moments and lets resolved
issues fade organically.

**Result:** Scored 81.0 vs independent's 96.0. Single rolling narrative was coherent and
well-written, but the model does better with 6 structured bullet points it can scan than
one paragraph it has to parse. Independent compression gives 6 chances to capture something
important; rolling gives one. The `format_highlights()` function was also optimized for
independent entries (open thread extraction, character indexes) which don't fire with a
single summary.

**Verdict:** Independent per-day compression confirmed as the stronger approach.

### Experiment: Cross-Concept Validation — TESTED 2026-03-14, PASSED

**Hypothesis:** The winning compression template (optimized on Jalapeño Corn Dog Bites) will
generalize to other recipe concepts without re-optimization.

**Results:**

| Concept | Baseline (no compression) | A/B (1 shot) | Optimized (full loop) |
|---------|--------------------------|-------------|----------------------|
| Jalapeño Corn Dog Bites | 83.8 | 81.6 | **96.0** (21 iter) |
| Mini Shepherd's Pies | 84.4 | 65.2 | **97.0** (21 iter) |
| Brown Butter Pecan Tassies | 76.4 | 75.8 | 77.6 (1 iter only) |

Mini Shepherd's Pies actually scored HIGHER than the original concept (97.0 vs 96.0).
Both Jalapeño and Mini Shepherd's converged in ~7 experiments with 3 keeps, auto-stopping
at 21 iterations. The convergence pattern is remarkably consistent.

Brown Butter Pecan Tassies only got 1 iteration but still showed improvement (+1.2).

**Verdict:** Template generalizes across recipes. Not overfit.

### Experiment: Compression Model Swap (Sonnet vs Haiku) — TESTED 2026-03-14, HAIKU WINS

**Hypothesis:** Sonnet (10x cost) produces better summaries than Haiku, which would mean
the bottleneck is compression quality rather than injection strategy.

**Results:**

| Compression Model | Baseline | Best (optimized, 21 iter) | Cost per compression |
|-------------------|----------|--------------------------|---------------------|
| **Haiku** | 83.8 | **96.0** | ~$0.003 |
| **Sonnet** | 63.2 | **95.0** | ~$0.03 |

Sonnet's summaries were more verbose and narrative ("Margaret opened with 'Jalapeño corn dog
bites. We're...'") vs Haiku's terser style ("Margaret questioned whether jalapeño corn dog
bites..."). The downstream dialogue model (Haiku) parses terse highlights more effectively.

Sonnet's baseline without compression was notably lower (63.2 vs Haiku's 83.8) — high
variance between runs makes single-shot comparisons unreliable, which is why we ran the
full 20-iteration loop.

**Verdict:** Haiku wins on both quality and cost. The bottleneck is not compression quality.

### Experiment: XML-Structured Injection — TESTED 2026-03-14, PLAIN TEXT WINS

**Hypothesis:** Wrapping compressed highlights in XML tags (`<week_context><day name="Monday">
<summary>...</summary></day>`) would help the model parse context more efficiently than plain
text bullets.

**Results:**

| Injection Format | Baseline | With Compression |
|-----------------|----------|-----------------|
| **Plain text** (bullets) | 83.8 | **96.0** (optimized) |
| **XML** (structured tags) | 83.8 | 72.6 → 79.8 (1 keep) |

Baselines matched (both 83.8), confirming the difference is real. Haiku treats XML tags as
formatting overhead rather than useful structure. The content is too short (~60 words per day)
for XML to provide parsing benefits.

**Verdict:** Plain text bullet injection wins. Content too short for XML structure to help.

## 14. Confirmed Winning Configuration (2026-03-14)

After 5 experiments across 2 sessions, the winning configuration is:

| Parameter | Winner | Alternatives Tested |
|-----------|--------|-------------------|
| **Compression model** | Haiku | Sonnet (95.0 at 10x cost) |
| **Compression mode** | Independent per-day | Progressive/rolling (81.0) |
| **Injection format** | Plain text bullets | XML tags (79.8) |
| **Context depth** | 12/8 (Fri-Sun), 8/4 (Mon-Thu) | Full context (87.0), 8/4 uniform (92.0) |
| **Saturday seeding** | Yes | No (92.0 vs 96.0) |
| **Best score achieved** | **97.0** (Mini Shepherd's Pies) | |

### Convergence Pattern

All optimization runs show remarkably consistent behavior:
- Best score found by experiment 5-7
- 3 keeps out of ~21 experiments (15% acceptance rate)
- Auto-stop triggers at iteration 20 checkpoint
- Trend: climbing → oscillating → stop

This convergence pattern held across different concepts, compression models, and injection formats.
